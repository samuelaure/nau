# Phase 9: CI/CD Hardening, Rapid Deployment & Integration Monitoring

## Objectives

1. **Strengthen CI/CD Pipeline (Disk Exhaustion Prevention):** Heterogeneous deployment on an 80GB (or 40GB) SSD requires strict disk management. We must refactor the GitHub Actions deploy script (`deploy.yml`) to systematically and safely execute `docker system prune` and targeted cleanup before/after pulling massive images to prevent "No space left on device" errors without losing database volumes.
2. **Accelerate Deployment Workflow:** Current deployment takes ~10 minutes. We must optimize the `deploy.yml` pipeline by utilizing advanced Docker BuildKit caching (e.g., inline, registry caching, or GitHub actions cache layer), skipping redundant steps, and possibly evaluating path-based filtering to only build if codebase files change (bypassing full rebuilds for simple documentation updates).
3. **Integration Observability & Testing:** Flownaŭ is connected to the naŭ Platform (`nauthenticity`, `9naŭ`) and external APIs (`OpenAI`, `Groq`).
   - We must establish robust health-check wrappers around these boundaries.
   - We must document integration points systematically.
   - We must write integration tests using mocks to verify that Flownaŭ gracefully handles API downtime or malformed inputs without cascading state-machine corruption.

## Tasks

### 1. CI/CD Disk Hardening

- [x] Refactor `deploy.yml` server SSH command block. Order of operations must be: stop older image version, `docker image prune -a -f --filter "until=1h"` (or similar safe prune), pull the new image, then `docker compose up -d`.
- [x] Add disk space diagnostic commands (`df -h`) before and after deployment in the CI logs to establish observability.

### 2. Accelerated Deployment

- [x] Implement Docker layer caching using `type=gha` (GitHub Actions cache) inside `docker/setup-buildx-action` and `build-push-action`.
- [x] Ensure that `npm ci` is effectively cached in the initial pipeline steps.
- [x] Explore separating the `Dockerfile.renderer` and `Dockerfile` logic to only build what explicitly changed, or rely entirely on layer caching.

### 3. Integration Testing & Control

- [x] **Document Integrations:** Create `docs/integrations.md` detailing exact webhook signatures, token passing rules, and fallback states for Naŭ APIs (InspoBase, 9nau Captures) and LLMs.
- [x] **Establish Boundary Fault-Tolerance:** Review `src/modules/ideation/sources/external-source.ts` and `src/modules/composer/scene-composer.ts`. Ensure failures from 3rd-party fetch requests are properly wrapped and cleanly bubbled up so crons don't crash but rather retry or mark items correctly.
- [x] **Mocked Tests:** Write missing Vitest test suites specifically mocking out `OpenAI`, `Groq`, and the Naŭ Inter-Service fetch requests to ensure system determinism under network failure conditions.

## Verification Criteria

- [ ] CI/CD logs show `df -h` and confirm disk footprint does not grow unboundedly with every push.
- [ ] Deployment time is reduced from ~10m down to under ~5-6m due to successful cache utilization on unchanged layers.
- [ ] `vitest` suite reports coverage over critical External API boundaries, appropriately handling timed-out fetch mocks.
- [ ] `docs/integrations.md` comprehensively maps external boundaries.
